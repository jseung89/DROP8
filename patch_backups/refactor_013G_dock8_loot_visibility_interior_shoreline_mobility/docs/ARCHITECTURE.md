# 아키텍처

## 구성

- `client`: Phaser 렌더링, 입력, HUD, 채팅 UI
- `server`: Colyseus Room, 권위 게임 루프, AI, 전투, 자기장
- `shared`: 무기 수치, 맵 상수, 타입, 검증 함수

개발 모드에서는 Colyseus의 Express 앱에 Vite 미들웨어를 붙여 HTTP와 WebSocket을 모두 2567 포트에서 제공합니다. 프로덕션에서는 `client/dist`를 같은 서버가 정적으로 제공합니다.

## 서버 권위 모델

클라이언트는 이동 방향, 조준 각도, 발사·획득·채팅 요청만 전송합니다. 서버가 위치, 벽 충돌, 탄약, 발사 간격, 총알, 피해, 방어구, 아이템, 사망, AI, 자기장과 우승을 계산합니다.

## 실행 주기

- 서버 시뮬레이션: 30Hz
- 상태 패치: 약 20Hz
- 클라이언트 렌더링: 브라우저 프레임 주기

현재 클라이언트는 서버 상태 사이의 단순 시각 갱신을 사용합니다. 본격적인 인터넷 경쟁 플레이를 위해서는 입력 재조정과 보간 버퍼를 더 정교하게 발전시켜야 합니다.

## Refactor 013 공간 아키텍처

`MapConfig`가 작은 맵, 큰 맵, 8번 부두의 정적 지형을 소유합니다. 맵 구현은 `shared/src/maps/`의 타입, 방, 물, 기존 큰 건물과 Dock 8 데이터로 분리됩니다.

공간 판정 계층:
1. 좌표에서 `buildingId + roomIndex` 계산
2. `SpacePortal`을 통한 방 연결과 실제 선분 시야 계산
3. 기존 벽·연막·수풀 판정 적용
4. 서버가 상호작용, AI 목표와 vault를 동일 공간 규칙으로 검증

강은 `RiverBand` 중심선과 구간별 폭으로 표현하며 `LandCrossing`이 다리·여울의 육지 판정을 덮어씁니다. `ShoreExit`은 높은 제방을 통과하지 않는 입수·상륙 구간과 AI 도하 노드를 제공합니다. 정적 데이터는 네트워크에 복제하지 않고 `roomIndex`와 `isSwimming`만 동기화합니다.

### 상태 크기 측정
8명, 전리품 예산, 오토바이 상태를 채운 Schema 전체 인코딩 모형에서 small 약 6.9KB, large 약 10.7KB, dock8 약 12.6KB였습니다. 정적 맵 데이터는 이 수치에 포함되지 않으며 네트워크로 반복 전송하지 않습니다. 서버의 기존 64KiB Encoder 안전 한도를 유지합니다.
